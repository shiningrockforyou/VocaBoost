## WORK ITEM A — Can we set a review-quiz FLOOR? Is a setting already implemented?

**Short answer: a review-quiz-SIZE setting exists (`reviewTestSizeMin`/`Max`), but there is NO per-class review-SCORE /
throttle floor — those thresholds are hardcoded. And there's an open question whether `reviewTestSizeMin` is actually WIRED.**

### What exists (verified)
- **`reviewTestSizeMin` / `reviewTestSizeMax`** — per-class-assignment settings, settable in the teacher UI
  (`AssignListModal.jsx:14-15`, defaults min=30/max=60), persisted (`db.js:796-820, 911-924`). These are meant to be the
  review-quiz size floor/ceiling.
- **Review-quiz size is computed** by `calculateReviewCount(recentSessions, reviewCap)` (`studyAlgorithm.js:223`):
  - `scoreBased = REVIEW_COUNT_BASE * (1.5 - avgLast3ReviewScore)` — lower scores → BIGGER review quiz.
  - bounded: `max(REVIEW_COUNT_MIN, min(scoreBased, reviewCap))`.

### The gap / question for Codex
- `calculateReviewCount` floors on a **hardcoded `REVIEW_COUNT_MIN`** (=15), and takes only `reviewCap` (the max). **Is the
  per-class `reviewTestSizeMin` actually plumbed into the floor, or is it stored-but-unused?** (`studyService.js:442` is
  the call site — confirm what `reviewCap`/min are wired from.) If unused, a teacher setting `reviewTestSizeMin=30` has no
  effect (the real floor is 15). **← Codex: verify wiring.**
- **There is NO per-class review-SCORE floor.** The throttle thresholds are **hardcoded** (`foundation.js:687-688` +
  `forcedPathway.js:37-38`: enter <0.30, exit >0.50) — not a class setting. So "a floor so a low-scoring student can still
  progress / isn't throttled below X" does **not** exist today; it would be a new capability. **← Codex: is a score-floor
  the right lever, or is size-floor what's wanted?**

## WORK ITEM B — Why is the grader faulty? (rejects correct Korean)

**Root cause (confirmed vs SUPPORT_RUNBOOK CS-2026-07-06b + CS-2026-06-29 B + the live prompt): the grading model
over-applies Rule 1 ("Restating the word") to correct KOREAN direct translations.**

### Facts
- **Model:** `claude-haiku-4-5-20251001`, temp 0.1 (`functions/index.js:1341`). (Note: CLAUDE.md says "OpenAI" — stale; it's Claude/Anthropic.)
- **The prompt** (`index.js:1267-1336`) is well-designed and lenient: Rule 1 examples are ENGLISH-only (run→running), and
  line 1277 explicitly says Korean near-synonyms + "answers matching the provided Korean definition" are CORRECT.
- **The failing cases** (CS-2026-07-06b, 김재민): `autobiographical ← 자전적인`, `indifferent ← 무관심한` — EXACT correct
  translations — rejected with reasoning literally *"just restating the word in Korean form — explain what it means."*
- **The 06-29 "fix" was too narrow.** It added "answers matching the provided Korean *definition* → CORRECT." But the
  failing answers are correct *translations of the WORD* that need not match the stored *definition string*, so they still
  trip Rule 1. Runbook flags it "incomplete/regressed."

### Diagnosis (WSL)
Rule 1's closing line — *"Rewriting the word in another form is not a meaning"* — is written for English inflections, but
the model generalizes it to Korean 1:1 translations (esp. morphological cognates: autobiographical↔자전적인). A precise
Korean translation IS the meaning; the model wrongly treats it as "restating." Haiku (a small model) applies the nuance
inconsistently.

### Candidate fix (grader prompt — owner-deployed code, not a data write)
1. **Scope Rule 1 to ENGLISH only:** "This rule applies ONLY when the answer is in English. A Korean answer is NEVER
   'restating' — a correct Korean translation, even a direct one-word literal one, IS the meaning and is CORRECT."
2. **Add positive examples:** `autobiographical ← 자전적인 → CORRECT`, `indifferent ← 무관심한 → CORRECT` (direct translation = meaning).
3. **Typo tolerance (1-char)** + adjective/verb-form leniency (partly covered by the 가난한 example; make explicit for 불협화음/교훈 typo cases).
4. Consider: a **stronger model** for grading (Haiku↔Sonnet), or a cheap **re-check pass** that re-grades only the
   rejections. **← Codex: model choice + whether prompt-only is sufficient, and a regression-test set (the runbook's
   known correct/incorrect examples).**
- Meanwhile: the per-answer **challenge** system (challengeStatus) is the student recourse; confirm TAs use it.
