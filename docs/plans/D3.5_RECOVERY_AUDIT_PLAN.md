# D3.5 — Pre-Migration Recovery & Adversarial Audit Plan (sandbox)

**Purpose:** before the irreversible D4/P5 migration, *prove the deployed fixes (PR-1/2/3 + D1–D3 cutover) actually
resolve July's failure modes* by (a) **seeding sandbox students into the EXACT broken states from the CS tickets** and
verifying they recover, and (b) **adversarially hammering the live UI** (button-spam, races, reloads, empty submits).
This is the full-UI prod audit that replaces the calendar soak — done while cutover rollback is still cheap.

## Terminology — the "CRITIC PASS" (bake this in; reuse the term)
**"Critic pass" = a completeness-critic convergence:** the same 5-way machinery we use for ground-truth convergence
(3 Fable agents + Codex + WinClaude, each independent, adversarial, never-trust-blindly, run as a convergence loop until
no surviving corrections), but pointed at a **plan** with the question *"what is MISSING or MIS-SCOPED here?"* rather than
*"is this fact true?"*. Distinct lenses: **Fable-1** = real-student roster completeness (does the cohort + named
students capture every July ticket? any ticketed uid with no seed?); **Fable-2** = scope correctness (fixed vs
observe-only misclassification; safety rails truly zero-26SM-write; tier-reweight soundness); **Fable-3** =
interaction/adversarial coverage (what button-spam / race / reload style is missing from Part B?); **Codex** = adversarial
failure-mode + assertion soundness (what corruption/edge is uncovered; will tier 3 actually catch client-side
regressions?); **WinClaude** = executor feasibility (can this be built + run on prod-Playwright? is `joinClass` the only
harness blocker? prod constraints?). WSL verifies every claim vs code/data, folds gaps, re-issues until convergence.

## Method (per scenario)
1. **Seed** the broken/precondition state on a 25WT student (Admin script — the *reverse* of the CS fix scripts).
2. **Drive** the action through the real UI (Playwright) or callable (emulator).
3. **Observe** — assert the UI behavior AND read back `class_progress`/`session_state`/`recentSessions`/`system_logs`.
4. **Verdict** — one of **PASS** / **FAIL** / **`INVALID_PRECONDITION`** (seed didn't match the broken state, UI didn't
   reach the action, or the assignment/threshold ≠ scenario contract → neither pass nor fail; re-seed, don't score).
   FAIL on a *fixed* family = regression → STOP + escalate. **Mitigation-canary** rows (A11/B17) = observe, **never STOP**.
   Every Tier-3 *fixed-family* scenario additionally requires a **server-path proof** (see Verdict handling).

**Three tiers, by what they actually prove:**
- **Tier 1 — emulator callable (local):** server logic given a *correct request*. ≈ a broad re-run of the D3 cert over
  all 156 real states + F edges. Cheap, ~1–2h.
- **Tier 2 — live-functions callable (admin-driven, sandbox users):** the *deployed* server + real Firestore recover the
  states. Still no client. ~1–2h. **(Under-claim note:** tier 2 is itself the FIRST-ever exercise of the deployed
  `FORCED_PATHWAY_ENABLED` hold-csd branch + `completeSession` in deployed legacy-write mode — real new certification
  value even if tier 3 slips.)
- **Tier 3 — live full-UI Playwright (prod, sandbox users):** the *real client routing + rendering + interactions*.
  ~1.5–2.5h wall-clock (the long pole).

**KEY INSIGHT (why the weighting).** Tiers 1/2 **bypass the browser client entirely** — they hand-build the payload and
call the function, so they prove the **server, NOT the UX**. Two consequences: (1) the cutover changed **client** flags
(client routing) — tier 1/2 can't test the thing the cutover actually changed; (2) most July tickets were **client/UX
symptoms** ("화면이 안 넘어감") even when the root was server-data — "Firestore has csd=6" ≠ "the dashboard shows Day 7 and
lets them proceed" (literally the 김우주 ticket). Client-side fixes tiers 1/2 are **blind** to: MCQ empty-submit guard,
retakeThreshold "93% shows fail", re-entry render, malform-loop, modals, frozen buttons, phantom display, and **"does the
screen actually advance."**

**DECISION (David-approved): reweight.**
- **Tiers 1/2 = a FAST, CHEAP PRE-FILTER** over all 156 real states (catch any server-data edge before spending UI time).
  NOT the pass/fail authority for UX.
- **Tier 3 = THE actual audit.** Expand its client-side/UX coverage beyond a per-family subset — assert the **client-side
  fixes** (empty-submit guard, re-entry render, threshold display, modals) and **"does the student's screen
  advance/recover,"** with real per-student UX breadth. **Pass/fail judgment on UX lives in tier 3.**

**Wall-clock (recommended):** build ~2–4h + run (tiers 1/2 background + tier 3 long pole) ~4–6h + triage/report ~1–2h ≈
**1–1.5 working days.** Max-exhaustive (all 156 through full UI) ≈ 2–2.5 days.

## Verdict handling
Four verdict classes:
- **FAIL on a PART A/A2/C/D "fixed" scenario** = a real regression on a previously-shipped fix → **STOP + escalate to
  David** (rollback candidate — the cutover is still reversible; this is exactly the window to catch it).
- **`INVALID_PRECONDITION`** (NEITHER pass nor fail) — the seed didn't reproduce the documented broken state, the UI
  never reached the intended action (`reachedTest=false` etc.), the seed didn't RENDER as intended (client re-derives via
  resolveListProgress/initializeDailySession — F-b), or the assignment/list/threshold ≠ the scenario contract (e.g.
  `retakeThreshold`≠0.92 on A11/B17, or a tier-1 run at the wrong flag posture). Re-seed; do NOT score. Prevents false-FAIL from a
  bad seed AND false-PASS from a scenario that never exercised its target.
- **PART B fail** = triage (product bug vs harness vs env).
- **PART E + mitigation-canary (A11/B17) = observe-and-document only, NEVER STOP.**

**SERVER-PATH PROOF (required on every Tier-3 fixed-family scenario)** — a UI scenario can pass via an unintended legacy
path and still show a good screen, so each must record ≥1 server-only signal: a server-emitted `system_logs` type
(e.g. `resolve_list_progress`, `review_recorded`, `csd_twi_reconciled` with `writtenBy=cloud-function`) OR a post-state
only the server path can produce. AND assert per run: **no canonical `list_progress` write while CANONICAL=false**; **no
client-created attempt marker while `SERVER_REVIEW_MARKER=true`**; **no client reset-delete while
`SERVER_RESET_PROGRESS=true`**. Without this, Tier 3 can go green while the student-facing complaint still reproduces.
