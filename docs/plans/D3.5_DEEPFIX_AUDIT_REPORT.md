**Status:** DRAFT for critic-pass convergence (8 entities) → then headless Fable+Opus max-effort confidence review.
**Date:** 2026-07-19. **Author:** WSL-Claude (coordinating WinClaude executor + Codex reviewer).
**Scope of truth:** every quantitative claim below was verified against live Firestore and/or deployed source; provenance

## 0. Executive summary

VocaBoost's daily-study progress engine had, through June–July 2026, produced a recurring stream of customer-support
(CS) tickets: students whose "current study day" (csd) jumped forward wrongly, got stuck in review-only mode, drifted
one day behind, or lost a saved test result. The root cause was structural: **progress was computed client-side**, so a
bad network moment, a double-click, an empty submit, or a stale browser session could write an incorrect durable state.
The **deep fix** ("deepfix") re-homed the authoritative progress computation to the **server** (Cloud Functions), behind
staged feature flags, so the durable record is derived from durable evidence (graded attempts) rather than client UI
state. That cutover (through stage **D3/P4**) is **live and certified**.

Before the final, **one-way** data migration (D4/P5 — moving the canonical progress store from `class_progress` to
`list_progress`), we ran the **D3.5 pre-migration recovery + adversarial audit**: we cloned real ticketed students'
*exact pre-fix broken states* into an isolated sandbox and drove them through the **live production UI** to prove the
deployed fixes actually recover them — plus a battery of synthetic adversarial states we invented.

**Result: 15 PASS / 0 FAIL** across every recovery family, each verified against Firestore ground truth and fresh
server-side proof. The audit also (a) caught and corrected three of our own tooling/fidelity defects before they could
produce false confidence, and (b) surfaced a *past* CS fix that had silently corrupted one real student's record — which
we then repaired. The deployed system is behaving as designed; the confidence is evidence-backed, not assumed.

## 1. Why the deep fix was necessary

### 1.1 The symptom stream (July CS tickets)
Real tickets clustered into a small number of **corruption families**:

- **Runaway day-jump (#16).** A student submitting empty or very-low-score reviews in quick succession could rocket csd
  forward. Canonical case **오윤권**: 8 reviews for days 5→12 submitted in **8 minutes** (scores 0/0/0/0/0/10/7/17), racing
  csd to 12 while his genuine progress was day 4. Each empty review "completed" a review-only day under the then-live
  client logic, so the day counter advanced without any real learning.
- **Throttle deadlock (#11-adjacent).** Students with poor review performance are (by design) throttled into a
  review-only mode until they demonstrate recovery. A bug/edge could leave them unable to escape — stuck reviewing with
  no path back to new words.
- **Off-by-one.** A student's csd could display one day behind their true completed day (a reconciliation lag).
- **Lost-save.** A new-word test result that failed to persist (network/grading failure) left the student stuck: the app
  believed they had not passed the day, blocking progression. Canonical cases: **황정민** (Base Camp Day 6), **최도훈**
  (Base Camp Day 16 — see §5, a special list-end variant).
- **List-end wall.** Students who finished a list hit a review-only "wall" and needed to be moved to their next list.

### 1.2 The structural root cause
All of these shared one root: **the client computed and wrote authoritative progress.** That means UI-level events —
double-clicks, retries, empty submits, stale/rehydrated sessions, races between a grade write and a navigation — could
each mutate the durable `class_progress` doc. The server had no veto. So the *same* underlying event (e.g. "submit a
review") could produce different durable outcomes depending on timing and client state, and there was no single
authority to reconcile them.

### 1.3 The fix: server-authoritative progress (staged)
The deepfix moves the authoritative computation server-side, behind flags, in stages:

- **PR-1** (`59df732`) — off-by-one reconciliation correctness: csd advances **on genuine completion**, not on a bare
  load or a bare score.
- **D2/P3** (`26cd8ee`) — 7 feature flags gating the server path.
- **PR-3 client** (`d2bb2bc`) — the "hold-csd" behavior: an empty/skipped review, or a throttle review-only day, records
  the review **without advancing** csd/twi (kills the #16 runaway).
- **P4 functions** (`0ddbb34`) — activate `FORCED_PATHWAY_ENABLED` server throttle with hysteresis + a grandfather epoch.
- **P4 client** (`6bffe1c`) — route progress / challenge / review-marker / reset **through the server**.
- **Docs/evidence** (`e20b532`).

**Live now:** client `6bffe1c` / functions `0ddbb34`. The server derives progress from durable graded attempts; the
client's saved session (`session_states`) becomes a **non-authoritative display cache** (a key property the audit
confirmed — see §4.2).

### 1.4 Why an audit *before* D4/P5
D3/P4 was already behaviorally certified in an emulator (10/10 on pinned `0ddbb34`). But:
1. The emulator/callable tiers **bypass the browser** — they prove the server, not the *user-visible* behavior, and most
   July tickets were client/UX symptoms.
2. D4/P5 is a **one-way** migration (canonical store moves `class_progress → list_progress`). Once done, rollback is
   expensive.
So we validated the deployed fixes against **real broken states through the real UI** while rollback is still cheap.

## 2. How the audit was run (methodology)

### 2.1 The three-tier model (and why tier 3 is *the* audit)
- **Tier 1** — emulator. **Tier 2** — live callable Cloud Functions. Both bypass the browser → they pre-filter for
  server correctness but cannot exercise the client routing the cutover changed.
- **Tier 3** — **live-production Playwright UI drives.** This is the real audit: it logs in as a seeded student, navigates
  the actual deployed app, takes the tests, and reads back the durable result. UX pass/fail lives here.

### 2.2 Faithful state reproduction (the core idea)
For each ticketed family we **cloned a real student's exact pre-fix broken state** into the sandbox — not an
approximation. `scripts/audit/clone-ticketed-prefix.mjs` reads the real 26SM student READ-ONLY and writes a sandbox
replica: full id-rewrite (studentId / classId / teacherId / testId), the real attempts (trimmed to the pre-fix day via
`maxStudyDay`), the real `class_progress` with a pre-fix overlay, and — critically — the reconstructed **session states**
(the corrupted saved sessions that were part of the failure). Fidelity was treated as the linchpin: an audit that seeds
an *easier* state than reality produces a "pass" that means nothing (we hit exactly this and fixed it — §4.5).

### 2.3 Sandbox safety (zero 26SM writes)
Every seeding and drive runs under a hardened guard (`scripts/audit/sandbox-guard.mjs`): sandbox class ids are
`25WT…`/`25WTsynth…`, underscore-free (so the `testId` tokeniser can't collide with real ids); `assertWrite` is
**fail-closed**; every run emits a safety artifact recording `writesToNonSandbox`. Across the entire audit that number
was **0**. Real-cohort (26SM) access was **read-only** for fidelity analysis, with one exception explicitly authorized by
the product owner (the 최도훈 data repair, §5).

### 2.4 The coordination loop (and the "never trust blindly" discipline)
Execution ran as a baton-file loop across three coordinated agents:
- **WSL-Claude** (planner/verifier — no push/Firebase creds): seeds states, writes verdict logic, verifies every result
  against Firestore.
- **WinClaude** (Windows executor — holds Playwright + Firebase + push creds): runs the live-UI drives.
- **Codex** (independent reviewer).
Standing rule, enforced throughout: **verify every claim against code or live evidence — including our own tooling's
output.** This was not ceremonial. WinClaude's in-session read-backs were premature **three separate times** (it reported
"no anchor," "no recovery," "orphaned/manual-pass-needed"); each time an independent Firestore read showed the async
write *had* landed and the recovery *was* working. None of those wrong conclusions entered the record.

### 2.5 Reaching and driving the tests (the hard part of tier 3)
A real blocker: the harness could not initially **reach** a new-word test (the 80-card study phase). We solved it from
source (not by trial-and-error):
- The first-run "Start Studying" customize modal must be dismissed (always present on a fresh browser context).
- The **kebab menu → "Skip to Test" → "Start Test"** path routes to the separate `/mcqtest` or `/typedtest/<class>/<list>`
  route. (Alternatively, pressing **`C`** per card *dismisses* it; the ChevronRight "Next card" arrow only *cycles* — that
  was the ~30s/card trap.)
- **Test mode varies per class** (MCQ vs typed) — a real product variable the harness must detect. The typed grader
  legitimately accepts the Korean definition (the test shows the English word and asks "Type your definition").
- **Direct-navigation standing rule** (learned the hard way): navigate straight to `/session/<class>/<list>` and assert
  the loaded list — the dashboard's list-picker can route a multi-list student to the *wrong* list (this caused one early
  false result, §4.5).

### 2.6 The verdict engine (server-path proof, M7)
`scripts/audit/assert-recovery.mjs` reads a driven student back and emits **PASS / FAIL / INVALID_PRECONDITION** per
family. A "fixed-family" PASS additionally requires **server-path proof (M7)**: a *server-only* `system_logs` type
(`resolve_list_progress`, `csd_twi_reconciled`, `review_recorded`, …) written **fresh within that round's window** — so a
green UI that silently ran a legacy client path cannot read as PASS. It also asserts the universal invariant that the
canonical `list_progress` store stays empty pre-P5. (We found and fixed a real bug in this engine — §4.4.)

### 2.7 What "critic pass" means (plan validation)
The audit *plan itself* was hardened by the same 8-entity critic-pass convergence being applied to this report:
3 Fable + 3 Opus + Codex + WinClaude, each asked "what ticketed student / interaction style / failure mode are we
missing?", looping until convergence, gaps folded back in.

## 4. Behavioral findings & self-corrections

### 4.1 Throttle escape requires TWO good reviews (and why our first read was wrong)
The deployed throttle uses hysteresis (`deriveThrottleModeServer`, foundation.js): **enter** review-mode when the last-3
review average < 0.30, **exit** when > 0.50, hold in the dead-band. Our early drives showed escape after *one* good
review — which turned out to be a **seed artifact**: the seed left `reviewMode` unset, so the hysteresis started from a
non-throttled prior state, and one mediocre review (avg ≈ 0.49, in the dead-band) "persisted" the escaped state. A
**faithful** throttled student (with `reviewMode=true` and `interventionLevel=1.0` persisted, as real throttled students
carry) correctly stays held on one review and needs a second to cross 0.50. Re-seeded faithfully, the deployed behavior
is exactly right (r58, step-log proof: `readback_after_review_1: reviewMode=true`; `readback_after_review_2:
reviewMode=false`).

### 4.2 `session_states` is inert for rendering (a key safety property)
A source trace confirmed the render study-day and word-pool derive **entirely** from durable `class_progress.currentStudyDay`
reconciled against the `attempts` anchor; `determineStartingPhase` treats attempts as the **sole authority** and reads
nothing render-affecting from the saved session. Consequence: **a poisoned/impossible saved session cannot corrupt the
render or falsely advance progress.** This is why the deepfix is robust against stale-session bugs.

### 4.3 Reconciliation is non-demoting (`safeCSD = max(stored, resolved)`)
The resolver never writes csd *down*. Consequences the audit relies on: an inflated runaway does **not** self-heal (so CS
repair stays necessary — demonstrated by 오윤권), and an impossible csd exceeding durable is **flagged/quarantined**, never
force-reset.

### 4.4 We caught a false-confidence bug in our own verdict engine
The verdict engine first reported extra PASSes by crediting **stale** prior-round `system_logs` as "server-path proof."
Root cause: a default time window of 0 (accept-all) plus a composite Firestore query that hit a missing index and was
silently swallowed to empty. A direct Firestore probe (query by `userId`, filter timestamp in JS) exposed it; we fixed the
engine to require **fresh** proof scoped to the run window. Corrected verdicts matched the direct probe exactly.

### 4.5 We caught two seed-fidelity defects (and a class-collision bug)
- **Wrong-list routing.** A multi-list student's dashboard routed the harness to the wrong list → a meaningless early
  result. Fixed by the direct-navigation rule (§2.5).
- **Class collision.** The seeder minted sandbox class ids from a per-run counter that reset each run, so unrelated
  students were crammed into shared class docs. Per-student state stayed uid-isolated (results uncorrupted), but it was
  structurally unfaithful; fixed to mint a **unique class per student**.
- **Unfaithful throttle seed** (§4.1).
Each was found by verification, before it could mislead the conclusion.

## 5. The 최도훈 forensic (the audit found a *past* CS error)

The one transient FAIL was choi_a12 — a clone of real student **최도훈**. Rather than explain it away, we dug into his full
history (product-owner's instruction: "the list was never trimmed — dig into every CS fix and every previous state").

**Findings:** Base Camp (`RmNNkuLP`) is genuinely a **15-day / 1200-word list** — his own day-15 new attempt introduced
words 1120-1199, the last 80 (the `words` subcollection contains exactly 1200, indices 0-1199). He completed days 1-15
legitimately. **There is no day-16 new test.** His ticketed "lost-save on the Day-16 new test" was therefore a
misdiagnosis — it was a **list-end** situation. The **CS-2026-07-07 manual-pass** for "day 16" computed a word range from
`day × pace` (16 × 80 → words 1200-1279) **without checking the list length**, writing a `passed` anchor for **words that
do not exist** and inflating his `totalWordsIntroduced` to **1280 on a 1200-word list** (a silent `TWI > list`
corruption the old integrity sweep missed, because it compared to the class's *first* list, not this one).

**Crucially, the deployed system handles his real (list-end) situation correctly and *better* than the manual pass did:**
it advances csd on the list-end review with twi **capped at 1200** — no phantom, no inflation. So the audit's "FAIL" was
the *wrong judge* applied to a mislabeled scenario, and it uncovered a real (if minor) legacy data defect.

**Remediation (product-owner authorized):**
1. Reclassified choi_a12 as **list-end** → now PASS.
2. Repaired the live record: `twi 1280 → 1200`, phantom anchor deleted, `csd = 16` preserved; backup taken; integrity
   sweep clean before/after (logged as **SUPPORT_RUNBOOK CS-2026-07-19b**).
3. Hardened the tools: `manual-pass.mjs` now **refuses** a day whose word range exceeds the list; `data-integrity-sweep.mjs`
   now compares twi/anchors to **each doc's own list** and adds a `phantomAnchor` check; new `fix-phantom-anchor.mjs`.
4. Scope scan across all of 26SM: the phantom-anchor signature is **unique to 최도훈** (every other manual-pass was mid-list).

## 6. What students can expect now (live + validated)

1. **No more runaway day-jumps** — empty/low reviews HOLD; they no longer advance the day counter.
2. **Stuck-in-review students can escape correctly** — after **two** good reviews (not one), they get new words again.
3. **Off-by-one day counters self-correct** on completion.
4. **Lost saves auto-recover mid-list** — retake the missing test → advance, **no CS intervention**.
5. **Finishing a list is handled cleanly** — advance on review days, word count capped (no phantom inflation).
6. **A corrupted saved session can't corrupt progress** — the durable record is authoritative.
7. **Normal students progress normally** — one day at a time, no regressions.

**Honest caveats:** students *already* inflated before the fix still need a one-time CS reset (the fix stops new bleeding,
doesn't retroactively heal old). One edge layer — browser-storage (localStorage/nonce) corruption — is **not yet
exercised**. And the final one-way D4/P5 migration has **not** run (gated on fresh backup + 25WT rehearsal + approval);
it is an internal storage change students won't perceive.

## 7. Confidence statement (for adversarial review)

The claim under test: *the deployed server-authoritative fixes recover the real broken states that generated July's
tickets, and do so durably.* The evidence: 15/15 faithful recoveries through the live UI, each cross-checked against
Firestore + fresh server-only proof; three of our own tooling/fidelity errors caught and corrected before they could
inflate confidence; a legacy data defect found and fixed. **Known limits:** no browser-storage-layer test; runaway
containment is not self-healing; D4/P5 unproven (out of scope here). Reviewers should probe whether any PASS rests on an
unfaithful seed, a stale proof, or an untested interaction path — those are exactly the failure modes we hit and fixed,
so they are the right places to push.
