# Work Item A (refined) — Optional per-class **Review Pass Threshold** (retake-until-pass)

**Status:** INVESTIGATE + PLAN (David-directed 2026-07-19; do not build until Codex-verified + David go).
**Owner loop:** WSL + Codex only.
**Supersedes** the earlier "wire `reviewTestSizeMin`" framing — David's actual ask is a *score gate*, specifically:
> "Just force them to retake until a set percent, like new word test. an optional lever applied by teachers."

---

## 1. What David wants
An **optional, per-class** setting: when a teacher turns it on, the **review quiz** behaves like the **new-word
test** — a student who scores below the set percent must **retake the review until they hit it**, and the day does
**not** advance until they pass. Off by default → today's behavior is unchanged.

## 2. Finding — the review quiz literally cannot fail today
Grep-confirmed (`reviewPassThreshold` / `requiresReviewRetake` / `reviewPassed` return **nothing** anywhere in
`src/` or `functions/`). Both the live writer and the reconciliation reader hardcode review to pass:

| Site | Code | Effect |
|---|---|---|
| `functions/index.js:434` | `const passed = ctx.sessionType === "review" ? true : score >= passThreshold;` | Review attempt → **always `passed:true`** |
| `functions/foundation.js:2614` | `const newPassed = attempt.sessionType === "review" ? true : newScore >= passThreshold;` | Reconciliation reader agrees |
| `src/services/studyService.js:1901` (comment) | *"The review test is the 'final' test and always passes"* | Day-2+ gate checks **only** the new-word test |

The review score is recorded (→ `recentSessions` → the throttle's rolling average) but there is **no per-session
pass/retake gate** on it. A student can score 5% on review and the session still completes, provided the *new-word*
test passed.

## 3. The pattern to mirror — how the NEW-WORD test enforces retake-until-pass
This is the exact machine David wants copied onto the review quiz:

1. **Per-class setting `passThreshold`** (0–100, default 95) — stored on the assignment.
   - UI: `AssignListModal.jsx:10,136` (`passThreshold` field).
   - Storage: `db.js:793,816` (assign) + `db.js:935` (edit-validate).
   - Read into config: `testConfig.js:33,57-58` (`passThreshold` + `passThresholdDecimal`).
2. **Server computes `passed`** at submission: `score >= passThreshold` (`index.js:434`; threshold read at
   `index.js:354`). The `passed` flag on the attempt doc is **authoritative** for advancement.
3. **Client Day-2+ retake gate** (`studyService.js:1908-1925`): if `!reviewOnlyDay && newWordAttemptPassed !== true
   && newWordScore < threshold` → **`return { requiresNewWordRetake: true }`** — blocks the COMPLETE write and forces
   a retake. (Ordering is load-bearing: the gate runs *before* `saveSessionState(phase: COMPLETE)`.)
4. **Client retake UI**: `DailySessionFlow.jsx:1015` `handleNewWordTestRetake`, `:1955` `<RetakePrompt>`, re-entry via
   `:1632 handleReEntryRetake`.

## 4. Proposed lever — `reviewPassThreshold` (mirror, default OFF)
Add an optional per-class `reviewPassThreshold` (0–100; **0 / unset = OFF = today's always-pass**). When set > 0:
- Review `passed = score >= reviewPassThreshold` (instead of hardcoded `true`).
- On fail → **retake**: block session completion, route back to the review test, do **not** advance csd/twi, until
  the student's review score clears the threshold.

**Safety property:** when `reviewPassThreshold` is unset/0 the code paths are byte-identical to today (the ternary
falls back to the literal `true`). Nothing changes for the ~all classes that don't opt in.

## 6. The careful part — server authority on the advance path (Codex r36-corrected)
The **server** (`completeSession` in `foundation.js`) is authoritative on advance-vs-hold for all 824 students. A
client-only review gate could be bypassed. Codex verified the server plumbing and corrected my first design:

- **Server already has the assignment.** `completeSession` → `assertEnrolledAssigned(uid,classId,listId)`
  (`foundation.js:1296`) returns `{classData, assignment, teacherId}` and already uses it for
  `deriveDailyPace(assignment)`. So `reviewPassThreshold` reads off the **existing** `assignment` object — **no
  client-trusted plumbing needed.** Legacy `assignedLists` with no `assignments[listId]` → `assignment=null` → must
  resolve `reviewPassThreshold=0` (OFF), never fail. Also destructure it into `writeAttemptTxn` (`index.js:404-405`)
  for the `:434` pass computation. (Caveat: assignment is read pre-transaction, same as pace today — a concurrent
  teacher edit uses the pre-txn version; acceptable.)
- **NOT `fpHoldCsd`/`review_recorded` (my error — Codex corrected).** Folding a failed review into the throttle-hold
  writer would append it to `recentSessions`, polluting the rolling review average and making the throttle
  *self-reinforcingly stickier* — a silent support trap. Instead: a **separate, explicit `review_retake_required`
  gate**, placed **after** the day-guard + F-4 evidence check but **before** any CSD/TWI advance or marker write. It
  writes **nothing durable**: no CSD/TWI, no review marker, no `users/{uid}/sessions` history, **no `recentSessions`
  append** (optional `system_logs` event only). This mirrors the new-word model (the failed attempt exists; the
  session simply isn't completed). Client `recordSessionCompletionViaServer` maps the status to
  `{ requiresReviewRetake: true }`.
- **Exemption must be SPECIFIC, not blanket `reviewOnlyDay` (my error — Codex corrected).** A blanket exemption would
  also exempt **throttle allocation-zero days**, defeating the gate on exactly the weak students it targets. Correct:
  exempt `reviewOnlyReasons.listComplete` (list-end) + `reviewStudyResume` (#9-resume) + the empty/all-mastered case
  (no review score to compare) — but **do NOT exempt `allocationZero`** (throttle days stay subject to the gate)
  unless David explicitly decides otherwise. The server's `reviewOnlyReasons` is the authority.

## 7. Design decisions / risks (for David + Codex)
1. **Stuck-student escape.** This is a mastery gate — a weak student *can* get frozen on review. That's the intent,
   but we need (a) the existing teacher **manual-pass** (`scripts/cs/manual-pass.mjs` / teacher override
   `passed:true`) to release them, and (b) consider a "N failed retakes → flag to teacher" signal so freezes are
   visible, not silent.
2. **Which score gates?** The **day's review test score** (per-session), matching "like new word test" — *not* the
   rolling average (that's the separate throttle). Confirm.
3. **Throttle interaction.** The throttle (rolling <30% → hold, David-locked) and this per-session review gate are
   orthogonal and can both hold. Verify they compose without double-writing or deadlocking `recentSessions`.
4. **Default OFF must be byte-identical.** `reviewPassThreshold` unset/0 → every path falls back to the literal
   `true` / existing behavior. This is the rollout-safety guarantee; assert it in the plan's test matrix.
5. **Flag-gate it** (mirror `FORCED_PATHWAY_ENABLED` discipline) so it can ship dormant and flip per the deploy
   protocol.

## 8. Open questions for Codex — ANSWERED (r36, `codex_reviewpass_r36.md`)
- **(A) → separate gate.** Implement a distinct `review_retake_required` status after day-guard + F-4, before advance;
  no `recentSessions` append. (NOT `fpHoldCsd` — that pollutes the throttle average.) See §6.
- **(B) → YES, server already has it.** `assertEnrolledAssigned` returns `assignment`; read `reviewPassThreshold` off
  it. No client plumbing. Legacy null → 0/OFF. See §6.
- **(C) → unlimited retakes + teacher surfacing.** No hard cap (a cap either auto-advances → defeats the gate, or
  silently locks → stuck students). Count failed review attempts per `student/class/list/day`, stamp/surface after N;
  manual override is the release valve. Failed attempts persist as `passed:false` (attempt docs), NOT into
  `recentSessions`.
- **(D) → specific exemptions only.** Exempt list-end + #9-resume + empty-review; do NOT exempt throttle
  `allocationZero`. See §6.

### Decisions:
1. **#9-resume exemption → RESOLVED (r37): do NOT exempt.** David + WSL position, Codex conceded the policy point
   (`codex_reviewpass_r37.md`): a resume day is not a dead-end like list-end (a valid new-word anchor exists; the
   forward path is "retake until pass"), so exempting it only adds a branch + a silent gate-hole. Apply the gate
   wherever a real review test exists. The ONLY non-list-end carveout is the **no-score** case (empty/all-mastered
   review — nothing to grade), keyed on *absence of a real review test*, NOT on `reviewStudyResume`.
2. Should **throttle allocation-zero** days be subject to the gate (Codex default + consistent with #1: yes)? Or
   always complete regardless of review score? — David to confirm; leaning YES (apply uniformly).
3. **Failed-retake surfacing threshold N** (after how many same-day review fails do we flag the teacher)? — David.

## 9. Rollout
Flag-gated, default OFF, opt-in per class. Ships dormant; byte-identical to current prod when no class sets a
threshold. Regression matrix must include: unset → unchanged; set + pass → advance; set + fail → retake (no advance,
no `recentSessions` append); set + list-end review-only → still advances; set + throttle allocation-zero + review
fail → retake (gate applies); set + teacher challenge lifts below→above threshold → advances.

## 11. Reader-correctness invariant (Codex r37 — WSL-VERIFIED, load-bearing)
**The invariant:** once review attempts can be `passed:false`, *every* reader that today treats "a review attempt
exists / pairs with the anchor" as "the day is complete" becomes a **false-green** path — a failed review that reloads
gets read as COMPLETE and advances the student, silently defeating the gate. So:

> **Under `reviewPassThreshold > 0`, a review attempt is completion evidence only if `passed === true`.**

**Verified trap (the primary site):** `determineStartingPhase` (`studyService.js:312-321`) returns `COMPLETE` when
`dayNumber>1 && newTest?.passed && reviewTest` — where `reviewTest` (`:266-270`) is found by
`reviewPairsWithAnchor(a,newTest) && isCompletionEngaged(a)`, **with no `passed` check**. A failed review is still
"engaged," so it would satisfy this and false-complete on reload. (Confirmed by direct read 2026-07-19.) This is
*especially* visible on #9-resume because the failed review range-pairs to a cross-class new-word anchor.

**Reader sites to update (all gated on threshold-on; default-OFF = today's behavior since review stays `passed:true`):**
| # | Site | Fix |
|---|---|---|
| 11.1 | `studyService.js:312-321` `determineStartingPhase` COMPLETE branch (+ the `:266-270` pairing finder) | Require `reviewTest.passed === true` before COMPLETE under threshold-on |
| 11.2 | `db.js` `getReviewForDay` (client completion-pairing) | Same `passed===true` requirement under threshold-on |
| 11.3 | `foundation.js:1638-1644` `getReviewForDayServer` marker-suppression / reconciliation pairing | Do not accept `passed:false` review as paired-completion evidence under threshold-on |
| 11.4 | retake-required path | Must leave the session in `REVIEW_STUDY`/retake state — must NOT leave a failed review attempt that the next load reads as COMPLETE |

**Default-off byte-equivalence:** with `reviewPassThreshold` unset/0, newly written reviews are still `passed:true`
and legacy reviews predate the field → all four readers behave exactly as today. This preserves the §4 safety property.
